Full Stack Development Overview

This section provides documentation on practices, frameworks, and technologies for full stack development within IE BestBank.

Git Feature Branch Strategy

The Git feature branch strategy enables streamlined, isolated development, fostering efficient collaboration between team members. It helps maintain a clean main branch, ensuring only high-quality, thoroughly reviewed code is merged. This strategy encourages modularity, where each feature or bug fix is developed independently and does not interfere with the work of other team members.

Branch Policies

  • Pull Request (PR) Reviews: All PRs must be reviewed and approved by at least one other developer.
  • Passing CI Checks: PRs must pass all automated CI checks (e.g., tests, linting) before merging.
  • Status Checks: CI pipelines must confirm successful builds before PRs can merge.

Actions Triggered

Feature Branch Push:

  • Automatically triggers deployment to the development environment, allowing developers to test changes in a live-like setting.

Pull Request to main:

  • Deploys to both development and User Acceptance Testing (UAT) environments for controlled validation before moving to production.
  • Ensures QA engineers and stakeholders can verify functionality.

Push to main:

  • Triggers deployment to all environments: development, UAT, and production.
  • Includes safety mechanisms such as automated smoke tests and rollback capabilities to maintain system stability.

Configuration

  • Code Reviews: Enforced by GitHub branch protection policies to ensure peer collaboration and code quality.
  • GitHub Actions: Used for automated CI/CD pipelines, streamlining build, test, and deployment processes.

Workflow Example

Feature Branch Workflow

  1. Developer creates a feature branch (e.g., feature/add-login-form).
  2. After completing the feature, pushing the branch to GitHub triggers CI pipelines.
  3. The code deploys to the development environment for testing, running automated tests.

Pull Request Workflow

  1. Developer opens a PR from feature/add-login-form to main.
  2. CI pipelines validate the PR (tests, linting) and deploy to UAT.
  3. If checks pass and PR is approved, it merges into the main branch.

Main Branch Workflow

  1. Merging into main triggers deployments to dev, UAT, and production.
  2. Automated tests (e.g., smoke tests) validate production environment.

Key Benefits

  • Collaboration: Reduced conflicts and isolated development enhance teamwork.
  • Code Quality: CI pipelines and reviews ensure production-ready code.
  • Automation: GitHub Actions handle repetitive tasks, letting developers focus on features.

CI Workflow for Frontend

Install Dependencies

  • Installs all required frontend packages (npm, yarn).
  • Ensures a consistent development environment for all contributors.
  • Purpose: Guarantees all libraries/tools are available for building, testing, deploying.

Run Tests

  • Executes unit tests for functionality and code quality validation.
  • Checks for edge cases, input validation, and component rendering.
  • Purpose: Detects issues early and enforces quality pre-deployment.

Build Static Files

  • Transpiles and bundles JS, CSS, and assets into production-ready files.
  • Minifies code and optimizes assets for performance.
  • Purpose: Prepares a lightweight, optimized application for deployment.

Deploy

  • Dev Environment: Deploys static files for testing and debugging.
  • UAT Environment: Deploys to UAT server for QA and stakeholder validation.
  • Prod Environment: Deploys final, optimized files to production or CDN for end-users.

Deployment Process by Environment

  • Development (Dev): Focus on feature validation and debugging.
  • UAT: Pre-production validation by QA/stakeholders.
  • Production (Prod): Final deployment, ensuring performance and reliability.

Purpose of Frontend CI Workflow

  • Automates repetitive tasks (testing, building, deploying).
  • Ensures consistency across environments.
  • Increases reliability by catching issues early.
  • Enhances efficiency by reducing manual intervention.

CI Workflow for Backend

Install Dependencies

  • Installs required backend libraries (e.g., pip install for Python).
  • Ensures consistent environment for all contributors.
  • Purpose: Guarantees necessary libraries for building, testing, running backend app.

Run Lint Checks

  • Checks code quality, syntax, and standards (e.g., flake8, black, pylint).
  • Purpose: Maintains consistency, readability, and reduces bugs from poor formatting.

Execute Unit Tests

  • Runs unit tests (e.g., pytest) to verify component functionality.
  • Generates coverage reports for completeness.
  • Purpose: Detects bugs early, ensures core features behave as expected.

Execute Postman Tests

  • Runs API tests for endpoint behavior and response accuracy.
  • Verifies HTTP status codes, payloads, error handling.
  • Purpose: Confirms endpoint correctness in real-world scenarios.

Build Docker Image

  • Creates a production-ready Docker image of the backend.
  • Tags images for versioning and easy rollbacks.
  • Purpose: Provides a consistent, portable deployment artifact.

Deploy

  • Dev: Deploy container for integration testing and debugging.
  • UAT: Deploy for QA validation and stakeholder approval.
  • Prod: Deploy final container ensuring scalability and availability.

Purpose of Backend CI Workflow

  • Automates testing and deployment to minimize manual effort and errors.
  • Ensures scalability via Docker images and container deployments.
  • Maintains reliability with rigorous testing and validation.
  • Enhances quality by enforcing coding standards, extensive tests, and API correctness.

Test Driven Deployment Strategy

Test-Driven Development (TDD) for user stories ensures that development is guided by clear, measurable acceptance criteria. This approach:

  • Drives Development: Focuses on fulfilling well-defined acceptance criteria for each user story.
  • Ensures Robust Validation: Addresses error handling, security checks, and proper input validation from the start.
  • Encourages Maintainability: Writing tests first leads to code that is modular, testable, and easier to maintain.

User Stories with Acceptance Criteria

User Story: Account Creation

Description: As a prospective bank customer, I want to register using a registration form (username, password, confirmation) to create a new bank account.

Acceptance Criteria:

  • Tests confirm presence of required fields (name, username, password, confirmation).
  • Tests validate input formats (e.g., password strength).
  • Error messages displayed for missing or invalid fields.

Test Example:

Input: Missing password confirmation field
Expected Result: "Password confirmation is required"

User Story: Secure Login System

Description: As a bank user, I want to log in with username and password to access my bank accounts securely.

Acceptance Criteria:

  • Tests ensure passwords are encrypted before storage.
  • Only correct username-password combos allow login.
  • Invalid logins return appropriate error messages.

Test Example:

Input: Correct username, incorrect password
Expected Result: "Invalid credentials"

User Story: User Interface for Transactions

Description: As a user, I want a straightforward UI to navigate and deposit into my account easily.

Acceptance Criteria:

  • Tests ensure intuitive UI with all key actions (deposit, view balance) visible.
  • Deposit inputs validated and processed correctly.
  • Error messages for invalid inputs (e.g., negative amounts).

Test Example:

Input: Deposit amount = -50
Expected Result: "Invalid amount"

User Story: Money Transfer

Description: As a bank user, I want to transfer money to other accounts by entering the recipient’s account number and amount, ensuring I don't exceed my balance.

Acceptance Criteria:

  • Tests validate transfer inputs (account number, amount).
  • Transfers do not exceed sender’s available balance.
  • Error messages for invalid or incomplete inputs.

Test Example:

Input: Transfer amount exceeds available balance
Expected Result: "Insufficient balance"

User Story: Multiple Accounts

Description: As a bank user, I want the ability to create multiple bank accounts and access them all.

Acceptance Criteria:

  • Tests confirm users can create/manage multiple accounts under one profile.
  • Tests validate viewing all linked accounts.

Test Example:

Input: Create a second account for an existing user
Expected Result: A second account is created and displayed

User Story: Default Admin Account

Description: As a system admin, I want a default admin account with predefined credentials to log in and access the user management portal.

Acceptance Criteria:

  • Tests verify a default admin account exists.
  • Only the default admin can access the admin portal initially.

Test Example:

Input: Login attempt with invalid credentials
Expected Result: "Invalid credentials" error

User Story: Admin Dashboard

Description: As a bank admin, I want a management dashboard to view all user accounts.

Acceptance Criteria:

  • Tests ensure admin dashboard lists all user accounts.
  • Tests confirm filtering and sorting capabilities.

Test Example:

Input: Admin requests to view all users
Expected Result: List of user accounts displayed

User Story: User Management by Admin

Description: As a bank admin, I want to list, create, update, and delete users and their passwords.

Acceptance Criteria:

  • Tests verify CRUD operations on user accounts.
  • Tests confirm proper permissions and security checks.

Test Example:

Input: Delete a user without proper admin auth
Expected Result: "Access denied"

Functionalities and Code Descriptions

General Functionality

  • Hello World Endpoint (/): Health check endpoint returns "Hello, World!".
  • Skull Endpoint (/skull): Debug endpoint exposing DB connection details (development use).

Account Management

  • POST /accounts: Creates a new bank account.
  • GET /accounts: Retrieves all bank accounts.
  • GET /accounts/<id>: Fetches details of a specific account.
  • PUT /accounts/<id>: Updates account details.
  • DELETE /accounts/<id>: Deletes an account by ID.

User Management

  • POST /users/register: Registers a user with username and password confirmation.
  • POST /users/login: Authenticates user credentials.
  • GET /users/<id>: Retrieves user details and associated accounts.

Transaction Management

  • POST /transactions: Executes money transfers between accounts, validating inputs and balances.
  • GET /transactions: Lists all transactions.
  • GET /transactions/<user_id>: Retrieves transactions for a specific user.

Deposit Management

  • POST /deposit: Deposits funds into a specified account, enabling balance testing.

Admin Routes

  • POST /admin/users: Admin creates new users with default accounts.
  • PUT /admin/users/<id>: Admin updates a user’s username/password.
  • DELETE /admin/users/<id>: Admin deletes a user and their accounts.
  • GET /admin/users: Retrieves all non-admin users.

Utility Functions

  • format_account(account): Returns formatted JSON for account objects.
  • format_user(user, accounts): Returns formatted JSON for user objects with accounts.
  • format_transaction(transaction): Returns formatted JSON for transaction objects.

Functional Tests Overview

The Postman API Collection ("Improved BestBank API Collection") validates the core functionalities of the system in Development, UAT, and Production. Environment variables like {{base_url}} allow dynamic testing across environments.

Key Functional Test Scenarios

General Tests

  • Hello World Endpoint: Verifies backend accessibility and returns "Hello, World!".
  • Skull Endpoint: Checks database connection details exposure in debug mode.

Accounts

  • Create Account: Tests account creation with required fields, expects 201 status and account id in response.
  • Get All Accounts: Verifies retrieval of all user accounts, expects 200 status and array of accounts.

Users

  • Register User: Tests user registration with validation, expects 201 and user id in response.
  • Login User: Validates authentication, expects 200 and user details on success.

Execution in UAT Environment

Steps to Execute Tests

  1. Open the collection in Postman: Load the "ImprovedBestBankAPICollection.json".
  2. Configure environment variables: Set {{base_url}} to point to the UAT environment.
  3. Run the collection:
    • Manually in Postman UI.
    • Automate with Newman: newman run ImprovedBestBankAPICollection.json --environment uat-environment.json

Integration with CI/CD Pipelines

  • Functional tests are executed as part of the CI pipelines in GitHub Actions.
  • GitHub Status Checks block merging PRs if any test fails.
  • Postman/Newman integration triggers test execution after deployment to UAT, ensuring code quality before moving forward.

Purpose

  • Validate API Functionality: Ensures endpoints behave as expected under various scenarios.
  • Early Issue Detection: Identifies broken APIs or logic issues before production deployment.
  • Streamlined Testing: Automation and consistency across deployments enhance reliability and speed.

Release Strategy and CD Workflows

Release Strategy: Docker Build and Push

Build the Docker Image

The Docker image is built twice:

  • Tagged with latest: Always reflects the most recent image.
  • Tagged with a version: Derived from the image-version step for traceability.

Push the Docker Image

Both tags are pushed to the production Docker registry:

  • latest tag: Ensures the most up-to-date version is readily available.
  • Version-specific tag: Provides a fixed reference for deployments.

Testing Before Deployment

Changes are tested thoroughly in UAT and DEV environments before merging into the main branch. This ensures issues are caught early, providing a stable release for production.

Deployment to Production

Once testing is completed and changes are merged into the main branch, the Docker images are deployed to the Production environment.

This strategy ensures a consistent, reliable, and tested process for delivering containerized applications to production environments.

Inner Loop

Description: The inner loop focuses on rapid, iterative development performed locally by developers. This includes writing, debugging, and testing code on their machines before committing to the shared repository.

Workflow

  • Local Environment Setup: Developers use Docker to containerize their application, ensuring consistency across local and production environments.
  • Debugging and Testing:
    • Frontend: Run locally (e.g., npm run dev) for quick UI iteration.
    • Backend: Use Docker + VS Code for debugging backend services, simulating production-like conditions.
  • Testing: Execute unit/integration tests locally (e.g., pytest, Postman) before pushing code.

Tools Used

  • Docker, VS Code (with Docker/debugging extensions), Local testing frameworks (pytest, Postman).

Purpose

  • Ensures code works as intended in a local, isolated environment.
  • Reduces deployment issues by catching errors early.

Outer Loop

Description: The outer loop refers to automated CI/CD processes that handle building, testing, and deploying apps to shared environments, ensuring a seamless transition from development to production.

Workflow

  • Code Commit: Developers commit changes to feature branches.
  • CI Pipelines: Automated build and test (unit, integration, functional) processes. Backend is Dockerized; frontend is built into production-ready assets.
  • Deployment: Automatic deployment to dev, UAT, and production using containerized workflows (Kubernetes, etc.).
  • Monitoring: Automated smoke tests and monitoring tools ensure successful, regression-free deployments.

Purpose

  • Provides a consistent, reliable deployment process.
  • Automates manual steps, reducing errors and speeding delivery.

Tools Used

  • GitHub Actions, Docker, Kubernetes, Postman/Newman for API testing, Monitoring tools (Prometheus/Grafana).

Comparison of Inner Loop and Outer Loop

Aspect Inner Loop Outer Loop
Scope Local development & testing Automated CI/CD & shared environments
Environment Developer’s local machine Dev, UAT, Production environments
Tools Docker, VS Code, local testing GitHub Actions, Docker, Kubernetes
Speed Fast iteration Slower, comprehensive testing & deployment
Purpose Debug & validate changes locally Automated deployment & maintain code quality

Summary

The inner loop allows developers to quickly iterate and test locally, while the outer loop ensures automated, reliable deployments across shared environments. Together, they minimize errors and accelerate high-quality software delivery.

Environment Progression

  • Development Environment: For testing/debugging during feature development.
  • UAT Environment: Staging area for QA and stakeholder validation.
  • Production Environment: Final stage for end-user access.

Automated Deployment Process

  • Feature Branch Push: Deploys to Dev environment for isolated testing.
  • Pull Request to main: Deploys to UAT for QA review and acceptance.
  • Push to main: Deploys to Production automatically once changes are approved.

CD Workflow for Frontend

Steps

  • Build Static Assets: Use npm run build or yarn build for production-ready files.
  • Deploy Static Files to CDN: Distribute via AWS S3+CloudFront, Azure Blob+CDN, or Netlify for global reach and improved performance.

Purpose

  • Ensures a lightweight, optimized frontend.
  • Improves performance and reduces latency for end-users.

CD Workflow for Backend

Steps

  • Build Docker Images with Tagged Versions: Encapsulates backend and dependencies for portability.
  • Push Docker Image to Registry: Stores image in Docker Hub, AWS ECR, or Azure Container Registry.
  • Deploy Docker Container to Kubernetes: Kubernetes handles scaling, load balancing, and resilience. Deployment manifests (YAML/Helm) define container specs.

Purpose

  • Provides scalability, fault tolerance, and high availability for backend services.
  • Versioning facilitates rollbacks if needed.
