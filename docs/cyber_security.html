Cyber Security Overview

This section provides guidelines for securing systems, data, and networks within IE BestBank.

GitHub Hardening Strategy

Our GitHub hardening strategy is based on mediating risks that could compromise software security. The goal is to enable developers to produce the most secure software possible by preventing sensitive information exposure, code vulnerabilities, and insecure dependencies, and by enforcing code reviews on every push. This ensures that changes to codebases comply with previously established development standards.

To achieve this, various tools from GitHub and external providers were implemented across BestBank-fe, BestBank-be, and BestBank-infra.

GitHub Secret Scanning

Implemented to detect any accidental exposure of secrets in our repositories. This is vital, especially since our repository is public, to avoid the risk of API keys, passwords, or tokens being discovered and exploited by malicious actors.

Developers were instructed to ensure no secrets are pushed to GitHub from their local environments. As a result, no secrets were ever pushed onto any development or main branches.

Push Protection

GitHub secret scanning was enhanced with push protection, which blocks pushes containing secrets. This automates the prevention of human error since pushes with secrets are automatically blocked. Should a developer override the block, an alert is generated.

CodeQL Implementation

CodeQL was implemented in our frontend and backend repositories to detect unpatched code vulnerabilities at the code level. The review runs on pushes and pull requests to the main branch, as well as every Friday at 9:27 AM.

CodeQL helps identify issues like SQL injection and XSS, among other vulnerabilities. Continuous monitoring of CodeQL alerts ensures critical risks are minimized before they affect the production environment.

OSSF Scorecard

The OSSF Scorecard focuses on repository-level security, including dependency management, code reviews, and general security hygiene. It runs on every push to the main branch and every Tuesday at 5:39 PM.

By catching vulnerabilities early, the OSSF Scorecard helps prevent issues at the source and provides insight into the security practices implemented in frontend, backend, and infrastructure repositories.

Dependabot

Dependabot automatically detects and provides upgrades for dependencies used across our repositories. This is critical as many of our repositories (except infrastructure) rely on multiple dependencies. Keeping dependencies up-to-date helps prevent known vulnerabilities from persisting.

Dependency Reviews

Dependency reviews further enhance dependency management by providing developers with information about how changes in dependencies might affect the project's security. If critical vulnerabilities are found, merging is blocked until the issue is addressed.

CODEOWNERS

Implementing CODEOWNERS ensures that only authorized contributors can push changes. All code must be reviewed by designated code owners (the main maintainers for each repository) before it can be merged.

Default Ruleset for Main Branch

In addition to the policies above, a default ruleset was defined for the main branch, which is used for deployment:

  • All changes must be submitted via pull request before merging.
  • At least one codeowner review is required prior to merging.
  • Force pushes are blocked.
  • Dependency review for critical vulnerabilities must pass before merging.
  • Deletion of the main branch is restricted.

By implementing this GitHub hardening strategy, our repositories are continuously monitored for vulnerabilities, and potential issues are addressed before they can materialize. This approach ensures a robust security posture throughout the development lifecycle.

Secure Credentials Management Strategy

In our development and deployment phases we follow the principle of least privilege (as outlined in PA-7 of Azure best practices). Each entity should have access only to the privileged data essential for their tasks. Based on this, we have outlined our secure credentials management strategy.

We combine the strengths of both GitHub Secrets and Azure Key Vault to store credentials securely.

1. Variables to Store in GitHub Secrets vs. Managed Identities vs. Azure Key Vault

GitHub Secrets

We use GitHub Secrets for deployment-time credentials and parameters needed by our GitHub Actions workflows during the CI/CD process.

  • Azure Service Principal Credentials (AZURE_CREDENTIALS): Used by GitHub Actions to authenticate with Azure.
  • Admin Names and Passwords: Sensitive data used to access our application as administrators.

Why store these in GitHub Secrets?

  • Deployment-Time Use: Required by the CI/CD pipeline during deployment.
  • Secure Storage in GitHub: GitHub Secrets store sensitive data safely for workflows, allowing initialization of Key Vault without exposure.
  • Not Needed at Runtime: Credentials are not required after deployment; they’re used to bootstrap our environment.

Managed Identities for Enhanced Security

Managed Identities provide Azure services with an automatically managed identity in Azure Active Directory (AAD), eliminating the need for explicit credentials and reducing the risk of credential leakage.

  • Used for Database Credentials: Securely manage connection strings to PostgreSQL via managed identities.

Reasons for Using Managed Identities:

  • Elimination of Credential Management: No manual handling of credentials, reducing leakage risk.
  • Seamless Integration: Easily integrates with Azure services for secure, streamlined access.
  • Enhanced Security Posture: Relies on AAD for authentication, adhering to least privilege without exposing credentials as environment variables.

Azure Key Vault

We use Azure Key Vault for runtime secrets and credentials needed by the application while running in Azure.

  • Certificates and Encryption Keys: Future TLS/SSL certificates, cryptographic keys.
  • Container Registry Credentials: For pulling images from a private container registry at runtime.

Why Azure Key Vault?

  • Runtime Secret Management: Secure storage and controlled access to runtime secrets.
  • Integration with Azure Services: Seamless access via Managed Identities.
  • Centralized Secret Management: Manage, rotate, and audit secrets centrally, allowing future integration with enhanced security features.

Permissions Across Environments (Prod, UAT, Dev)

Following best practices for protecting secrets:

  • Apply Granular Access Control: Use Azure RBAC to control who/what can access secrets.
  • Leverage Managed Identities: Eliminates managing credentials in code.

Setting Up Permissions for Azure Key Vault

Use Azure RBAC for Access Control:

  • Key Vault Administrator: Full access to keys, secrets, certificates (e.g., service principal).
  • Key Vault Reader: Read metadata (no secret values).
  • Key Vault Secrets Officer: Manage secrets without altering Key Vault configuration.
  • Key Vault Secrets User: Applications accessing secrets at runtime.

Separate Key Vaults per Environment (dev, UAT, prod):

  • Isolate secrets and access controls per environment.
  • Environment-specific access: Grant access only to resources/identities in that environment.
  • Different permission models: Prod and UAT more constrained than dev, maintaining deployment integrity.

Data Flow Diagrams

High-Level DFD:

Actors: End-User → Frontend → Backend APIs → Database & Infrastructure Services

Trust Boundaries:
1: Between user’s browser and frontend.
2: Between frontend and backend APIs.
3: Between backend APIs and infrastructure.

Low-Level DFDs

1. User Authentication

Data Flow: User submits login credentials → Backend validates credentials against the database → Response returned to the user.

Threats:
Spoofing (unauthenticated access), Information Disclosure (credentials over insecure channel)

Mitigations:
Enforce HTTPS, Implement MFA.

2. Transaction Processing

Data Flow: User initiates a transaction → Backend validates and updates balances in database → Transaction recorded in logs.

Threats:
Tampering (manipulating transaction data), Repudiation (lack of audit trails)

Mitigations:
Use database transactions for atomicity, Enable detailed audit logging.

Security Implementation Requirements

Our project is built on the "assume breach" principle, presuming potential vulnerabilities and focusing on minimizing risks from the outset. From the planning stage, we implemented comprehensive cybersecurity strategies (utilizing Open SSF and SAFECode frameworks) to mitigate controllable risks during the development phase, enabling us to prioritize more complex threats as they arise later in the deployment phase.

SAFECode Practices Implemented

We chose five principles from the SAFECode “Fundamental Practices for Secure Software Development” and documented how we applied them: ( Reference)

Secure Design Principles (Based on Saltzer and Schroeder)

Economy of Mechanism

  • Simplified Architecture: The IE Bank app focuses on core functionalities (user, account, transfers) without unnecessary features, reducing potential vulnerabilities.
  • Minimal Dependencies: A simple frontend framework and limited third-party integrations keep the system lightweight and easier to audit.
  • Basic Authentication: Straightforward username/password avoids complexities and reduces security pitfalls that might arise from advanced methods (e.g. OAuth).

Fail-Safe Defaults

  • Role-Based Access Control (RBAC): Clear roles with explicit permissions ensure no user gets extra privileges by default.
  • Default Denial: All resources are inaccessible by default, only explicitly permitted actions are allowed.
  • Secure Defaults in Configuration: Firewall rules and DB permissions start from deny-all and are opened as needed.

Complete Mediation

All access to every resource is checked for authorization each time, ensuring no cached or assumed permissions.

Least Privilege

  • Granular Permissions: Admins handle only user-management tasks; regular users manage their own accounts. Each entity only gets the minimal permissions they need.
  • Service Account Restrictions: Service principals and DevOps accounts have minimal required permissions.
  • Database Access Controls: Application components can only query data they strictly need, reducing leakage risk.

Least Common Mechanism

  • Isolated Functional Modules: Admin and user portals are separate modules, limiting shared dependencies.
  • Separate Environments: Dev, UAT, and prod environments are independent, preventing cascading issues.
  • Independent Authentication Flows: Admin and user portals have distinct authentication, reducing common targets.

Psychological Acceptability

  • Intuitive UI: Simple interfaces encourage correct security behavior by users.
  • Clear Feedback: Immediate confirmations or error messages help users avoid mistakes.
  • Streamlined Authentication: Straightforward login/registration reduces user errors that could compromise security.

Compromise Recording

  • Comprehensive Logging: Critical events (logins, updates, transactions) are logged for forensic analysis.
  • Audit Trails: Administrative actions are fully traced.
  • Monitoring and Alerts: Real-time monitoring of logs detects suspicious activity, supported by secure log storage.

Use of Static Analysis Security Testing (SAST) Tools

We employed SAST tools to identify flaws early in the code:

CodeQL as SAST for Frontend and Backend

  • CodeQL Integration: Implemented for Vue.js frontend and Python backend.
  • Comprehensive Code Analysis: Deep semantic checks catch issues traditional linters miss.
  • Automated Detection: Runs on every commit/PR, ensuring continuous security assessment.
  • Custom Queries: Tailored queries for IE Bank’s needs enhance vulnerability detection.

GitHub Secret Scanning and Push Protection

  • Secret Scanning: Prevents committing API keys, passwords, and tokens.
  • Push Protection: Blocks secret-containing pushes, alerting developers if they attempt to override.

OSSF Scorecard

  • Security Metrics Evaluation: Measures adherence to security best practices.
  • Continuous Improvement: Insights guide SAST configurations and focus areas.

Dependabot and Dependency Reviews

  • Vulnerability Management: Keeps dependencies updated, reducing vulnerabilities from outdated packages.
  • Proactive Security: Reviews assess potential security risks from new or updated dependencies.

Establishment of Log Requirements and Audit Practices

Centralized Logging (Application Insights & Log Analytics)

  • Application Insights: Logs user interactions, errors, and performance for frontend and backend.
  • Log Analytics Workspace: Azure resources send diagnostics here, enabling unified monitoring and easier audits.

Defined Log Requirements Aligned with SLA and SLOs

  • SLA & SLO-driven Logging: Logs support measuring SLIs to ensure SLA compliance.

Secure Log Storage and Protection

  • Encryption and RBAC: Logs are encrypted and access is controlled.
  • Immutable Logs: Prevents tampering; ensures integrity for forensic analysis.

Audit Trails and Accountability

  • Detailed Metadata: Timestamps, user IDs, and action types enable traceability.

Automated Log Analysis and Monitoring

  • Azure Workbooks/Dashboards: Visualizes SLIs and key metrics in real-time.
  • Azure Alerts & ChatOps: Immediate Slack notifications for suspicious activities.

Integration with Incident Response

  • Slack Integration: Alerts posted to channels for rapid incident response.

Regular Auditing and Review

  • Scheduled Audits: Ensures ongoing compliance and continuous improvement.

Threat Modeling and Risk Acceptance Process

Threat modeling was performed regularly (every 1-2 PRs) by the cybersecurity team, identifying threats based on SAST results and logical reviews.

  • Risk Threshold: Severe risks fixed immediately, high risks fixed if feasible, medium/low deferred for next sprint.
  • Collaboration: Cybersecurity team, full stack devs, and product manager coordinate on prioritizing fixes.

Automated Testing

Test-Driven Development

  • Unit Tests: Positive and negative tests for backend (pytest) and frontend (Vue.js).
  • Functional Tests with Postman: End-to-end tests in UAT, integrated into CI/CD, must pass before PR approval.

OpenSSF Principles Implemented

Monitor Vulnerabilities in Dependencies

  • Dependabot & Dependency Reviews: Ensures outdated/insecure dependencies are quickly addressed.

Prefer Memory-Safe Languages

  • Backend (Python): Automatic memory management prevents common vulnerabilities.
  • Frontend (Vue.js/JS): Automatic memory operations ensure safer memory handling.

No Secrets in Repositories

  • GitHub Secret Scanning & Push Protection: Prevents credential leaks at the source.

Control Asset Loading from Own Domains

  • Minimized External Dependencies: No external JS/CSS/media directly loaded from uncontrolled domains.

Evaluate Open Source Software Dependencies

  • Dependency Vetting: Each dependency documented and researched before adoption.

Conclusion

By implementing principles from SAFECode and OpenSSF frameworks, utilizing SAST tools, enforcing secret scanning, maintaining secure logging and auditing, and practicing careful dependency management, we have laid a strong foundation of security. Our approach ensures minimal risk exposure throughout development and sets the stage for proactive threat mitigation during later deployment and operations phases.
