Cloud Architecture Design Overview

This section outlines strategies, patterns, and best practices for designing and managing cloud architectures at IE BestBank.

Infrastructure Architecture and Environments Design

The infrastructure for the IE BestBank MVP is designed to ensure scalability, security, and cost-effectiveness, adhering to the functional and non-functional requirements of the application. By following a cloud-native approach and leveraging Microsoft Azure services, we implement a robust DTAP (Development, Testing, Acceptance, Production) environment strategy. This design aligns with the Azure Well-Architected Framework and integrates industry best practices.

Key Azure Services and Architecture Components

Azure Static Web Apps for Frontend

  • Lightweight and cost-effective
  • Implementation of CI/CD Integration through GitHub Actions
  • Ease of scalability and faster delivery speed of frontend assets

Azure App Service for Backend

  • Highly reliable when hosting API services
  • Azure monitoring tools can be integrated easily into it
  • Effecient when it comes to conatiner mangament decisions

PostgreSQL Azure Database

  • Flexible Server mode utilization allows for higher performance and cost optimization

Azure Container Registry (ACR)

  • Basic SKU implemented for Development and UAT, Standard SKU for Production
  • Linked with an Azure Key Vault in order to ensure security of credentials

Azure Key Vault

  • Automated secret retrieval
  • Standard SKU across all environments enusring consistency/li>

Monitoring and Logging

  • Azure Log Analytics: PerGB2018 SKU for scalable logging/analysis, centralized for infrastructure.
  • Azure Application Insights: 90 day retention period, Sufficient telemetry data for long-term trends and live environment debugging.
  • Azure Workbooks: Provides us with a visualization of logs and metrics for actionable insights.
  • Azure Alerts: Real-time incident notifications.

Environment Setup: DTAP Strategy

IE BestBank's MVP employs a structured DTAP (Development, Testing, Acceptance, Production) strategy in order to improve and propelry ensure application lifecycle. The main reason behind as to why we deployed this strategy was to ensure that each stage operates independently, ensuring that errors or issues were not carried over to the next stage. This allowed us to ensure our stakeholders of the quality of the application before it was deployed to production, increasing its reliability and trust.

Advantages of the DTAP Strategy

  • Improved quality of the application
  • Increased reliability and trust
  • Reduced risk of errors and issues
  • Enhanced collaboration between development and operations teams

Testing/UAT Environment

  • Resource Group: BCSAI2024-DEVOPS-STUDENTS-A-UAT.

Production Environment

  • Resource Group: BCSAI2024-DEVOPS-STUDENTS-A-PROD.
  • Focus:
    • Ensures that the application is stable and reliable for end-users.

    • Deployments are triggered only after successful testing in the UAT environment.
    • Uses Azure Key Vault for secure secrets management.
    • High availability and scalability are ensured through Azure App Service and Azure Database.

Well Architected Framework Design

Operational Excellence Pillar

The Operational Excellence pillar focuses on running and managing the workload to deliver business value and to improve efficiency

Deployment Strategies

  • Continuous Integration and Deployment: Implemented CI/CD pipelines using GitHub Actions for automated deployment of code changes across environments. Ensured consistent deployment practices with automated testing and validation.
  • Environment Provisioning: Utilized Azure Bicep templates for infrastructure as code (IaC) to provision and manage resources consistently across environments.

Monitoring and Diagnostics

  • Monitoring Tools: Leveraged Azure Application Insights and Log Analytics Workspace for real-time monitoring and diagnostics.
  • Custom Metrics: Defined custom metrics and alerts to monitor application performance and resource utilization. These were based on our predefined Service Level Objectives (SLOs) and Service Level Indicators(SLIs).

Incident Management

  • Real-Time Notifications: Integrated Azure Logic Apps with Slack for instant incident notifications and alerts.
  • Automated Recovery: Developed automated recovery scripts to restore services in case of failures.
  • Documentation and Tracking: Maintained detailed incident reports and tracked resolution times for continuous improvement.

Continuous Improvement

  • Conducted regular retrospectives after sprints and implemented feedback loops to optimize workflows and enhance operational efficiency.

Cost Optimization Pillar

Focuses on optimizing costs without sacrificing performance by leveraging cloud services efficiently.

Cost-Efficient Infrastructure Provisioning

  • Infrastructure as Code (IaC): Utilized Azure Bicep templates for consistent and automated resource provisioning.
  • Right-Sizing Resources: Selected appropriate resource tiers based on workload requirements to balance performance and cost.

Cost Monitoring and Analysis

  • Utilized Static Web Apps for frontend to minimize costs and optimize performance, as well as PostgreSQL Flexible Server for cost-effective database operations.
  • Implemented environment-specific retention policies in our Application Insights and Log Analytics (30-90 days).
  • Integrated Slack with Azure Logic Apps for real-time cost monitoring alerts.

Automation for Cost Reduction

  • Auto-Scaling: Configured auto-scaling for Azure App Service and PostgreSQL to adjust resources based on demand, reducing costs during low-traffic periods.
  • Automated Shutdown: Scheduled automated shutdowns for non-production environments to minimize costs outside of working hours.
  • Cleanup of Unused Resources: Implemented automated scripts to identify and remove unused resources, reducing unnecessary expenses.

Performance Optimization Pillar

Focuses on optimizing performance to meet the demands of the workload and deliver a seamless user experience.

Resource Utilization

  • Utilized Azure App Service for hosting backend services, ensuring high availability and scalability.
  • Configured Azure Container Registry for efficient container management decisions.
  • Implemented Azure Key Vault for secure secrets management across all environments.

Cost-Effective Performance Improvements

  • Utilized Azure CDN for efficient content delivery and reduced latency, enhancing user experience.

Continuous Performance Optimization

  • Conducted regular performance testing and optimization to identify bottlenecks and improve system efficiency.

Enhancing System Efficiency

  • Implemented Azure Log Analytics and Application Insights for real-time monitoring and diagnostics to identify performance issues.
  • Defined custom metrics and alerts to monitor application performance and resource utilization, ensuring optimal system efficiency.

Security Pillar

Focuses on protecting data, systems, and assets from threats and vulnerabilities throughout the application lifecycle.

Design Features

  • Azure Key Vault: Centralized secrets management for secure storage and access control. Ensures sensitive data is encrypted and protected.
  • CI/CD Security Tools:
    • CodeQL: Static code analysis tool that identifies security vulnerabilities in the codebase.
    • OSSF Scorecard: Open Source Security Foundation tool that assesses the security posture of open-source projects.
  • Dependency Management:
    • Dependabot: Automated dependency management tool that detects and fixes vulnerabilities in dependencies.
  • Secrets Management:
    • GitHub Secret Scanning: Scans repositories for exposed secrets and alerts developers to take corrective action.
    • Push Protection: Prevents accidental exposure of secrets by blocking pushes that contain sensitive information.
  • Encryption: Utilized Azure Key Vault for encryption at rest and in transit to protect sensitive data. Protected data in transit using HTTPS and TLS encryption.
  • Threat Modeling: Conducted threat modeling sessions to identify potential security risks and implement appropriate controls.
  • Secure Coding Practices: Followed secure coding practices to prevent common vulnerabilities such as SQL injection, cross-site scripting (XSS), and others.

By incorporating these security measures, the application is protected against threats and vulnerabilities, ensuring data confidentiality, integrity, and availability throughout the system lifecycle.

Reliability Pillar

Focuses on ensuring system availability, fault tolerance, and resilience to maintain a consistent user experience.

Ensuring System Availability

  • High-Availability Targets: Defined very high level Service Level Objectives (SLOs):
    • 99.9% for Azure Key Vault (Standard tier).
  • Redundancy Strategies: Implemented redundancy at the infrastructure level with Azure App Service and Azure Container Registry to ensure high availability.

Resilience through Backup and Recovery

  • PostgreSQL Configuration: Configured automated backups and point-in-time recovery for PostgreSQL database to prevent data loss. Made use of PostgreSQL Flexible Server in Burstable configurations for cost-effective backup and recovery.
  • Soft Delete for Data Protection: Implemented soft delete functionality in the application to protect against accidental data loss.

Monitoring and Alerting

  • Real-Time Observability: Utilized Azure Monitor for real-time monitoring and alerting of application performance and resource utilization.
    • Application Insights with 90-day retention for application metrics.
    • Log Analytics Workspace with 30-day retention for operational logs.
  • Metric-Based Alerting: Implemented alerts for key performance indicators (KPIs), including:
    • HTTP response time thresholds.
    • Latency Analysis.
  • Incident Notifications: Integrated Azure Logic Apps with Slack for real-time incident notifications and alerts.

Fault Tolerance Mechanisms

  • Infrastructure Resilience: Implemented Azure App Service and Azure Container Registry for fault tolerance and high availability.
  • Graceful Degradation: Designed the application to gracefully degrade functionality during service disruptions to maintain user experience.

12-Factor App Design

Overview

The 12-Factor App methodology outlines best practices for developing modern, scalable, and maintainable cloud-native applications. The IE BestBank MVP application adheres to these principles, ensuring a robust and efficient architecture. By following these guidelines, we achieve a high level of operational excellence, security, and reliability in our cloud-native design.

12-Factor Principles Applied

1. Codebase

Maintain a single codebase tracked in revision control, with multiple deployments.

  • Implementation:
    • Source code is stored in a single GitHub organization made of 3 GitHub Repositories(Frontend, Backend, and Infra), enabling version control and collaboration.
    • CI/CD pipelines deploy the same codebase to multiple environments (Development, UAT, Production).
    • Infrastructure as Code (IaC) templates are version-controlled alongside the application codebase.

2. Dependencies

Explicitly declare and isolate dependencies.

  • Implementation:
    • Frontend and backend dependencies are explicitly declared in package.json and requirements.txt files, respectively.
    • Dependencies are containerized using Docker, ensuring consistent runtime environments across deployments.
    • Dependency versions are pinned to prevent unexpected changes and ensure reproducibility.

3. Configuration

Store configuration in the environment.

  • Implementation:
    • Environment-specific configuration settings are stored in Azure Key Vault.
    • Configuration settings are injected into the application at runtime using Bicep Parameter Files, allowing for easy environment-specific customization.
    • Secrets and sensitive information are securely managed and accessed through Key Vault integration.

4. Backing Services

Treat backing services as attached resources.

  • Implementation:
    • Backing services (e.g., databases, message queues) are treated as external resources and accessed via connection strings stored in Azure Key Vault.
    • Services are provisioned and managed using IaC templates, ensuring consistent and repro ducible environments.

5. Build, Release, Run

Strictly separate build and run stages of deployment.

  • Implementation:
    • Build and release stages are separated using CI Pipelines, ensuring that the build process is consistent across environments.
    • Container images are built and stored in Azure Container Registry, decoupling the build process from the runtime environment.
    • Release and deployment stages are automated using CI Pipelines, ensuring consistent and reproducible deployments across environments.

6. Processes

Execute the application as one or more stateless processes.

  • Implementation:
    • Frontend and backend services are designed as stateless processes, enabling horizontal scaling and fault tolerance.
    • Containerized applications are deployed as stateless microservices, allowing for independent scaling and deployment.

7. Port Binding

Use container ports to expose services.

  • Implementation:
    • Frontend and backend services are exposed through container ports, enabling communication with external services.
    • Containerized applications are deployed to Azure App Service, which manages port binding and routing automatically.

8. Concurrency

Use concurrency to improve responsiveness and throughput.

  • Implementation:
    • Frontend and backend services are designed to handle concurrent requests, improving responsiveness and throughput in high-traffic scenarios.
    • Containerized applications are deployed to Azure App Service, which automatically manages concurrency and scaling based on demand.

9. Disposability

Maximize robustness with fast startup and graceful shutdown.

  • Implementation:
    • Frontend and backend services are designed to start up quickly and shut down cleanly, minimizing downtime and improving overall system reliability.
    • Containerized applications are deployed to Azure App Service, which manages startup and shutdown processes automatically.

10. Dev/Prod Parity

Keep development, staging, and production environments as similar as possible.

  • Implementation:
    • Development, testing, and production environments are provisioned using the same IaC templates, ensuring consistency and parity across environments.
    • CI/CD pipelines deploy the same codebase to all environments, minimizing configuration drift and ensuring consistent behavior.

11. Logs

Treat logs as event streams.

  • Implementation:
    • Application logs are treated as event streams and stored in Azure Log Analytics for real-time monitoring and analysis.
    • Log data is collected, aggregated, and analyzed using Azure Application Insights, providing actionable insights and performance metrics.

12. Admin Processes

Run admin/management tasks as one-off processes.

  • Implementation:
    • Admin tasks, such as database migrations and backups, are run as one-off processes using Azure DevOps pipelines or Azure CLI scripts.
    • Scripts are executed in isolated environments to prevent interference with production services.

Summary

By following the 12-Factor App methodology, the IE BestBank MVP application achieves a high level of operational excellence, security, and reliability. The application is designed to be scalable, maintainable, and cloud-native, ensuring a seamless user experience and efficient development workflow.

Release Strategy

The release strategy for the IE BestBank MVP application is designed to ensure efficient and reliable deployments across multiple environments. By following a structured approach to release management, we can deliver new features and updates with minimal risk and downtime.

Environment-Specific Design

The application is designed to run in multiple environments, including development, testing, staging, and production. Each environment serves a specific purpose and is configured to meet the needs of its users. The following sections outline the key components and deployment triggers for each environment.

Development Environment

  • Purpose: Provides a sandbox environment for developers to test new features and changes.
  • Configuration: Uses a lightweight infrastructure setup to facilitate rapid development and testing.
  • Deployment Trigger: Commits to feature branches trigger deployments to the Development environment.
  • Setup:
    • Integrated with CI/CD pipelines for automated testing and deployment.
    • Containerized application components are stored in Azure Container Registry for easy deployment.

UAT Environment

  • Purpose: Acts as a staging environment for user acceptance testing and stakeholder reviews.
  • Configuration: Mirrors the Production environment to validate application functionality and performance.
  • Deployment Trigger: Pull Requests (PRs) to the main branch trigger deployments to the UAT environment.
  • Setup:
    • Deployments proceed only after successful testing in the Development environment.
    • Uses Azure Key Vault for secure secrets management and environment-specific configurations.

Production Environment

  • Purpose: Hosts the live application for end-users and customers.
  • Configuration: Configured for high availability, scalability, and performance.
  • Deployment Trigger: Pushes to the main branch trigger deployments to the Production environment afetr successful deploying to UAT Environment.
  • Setup:
    • Deployments are automated and require successful testing in the UAT environment.
    • Uses Azure Key Vault for secure secrets management and environment-specific configurations.

Key Components

Source Control

  • Git: Used for version control and collaboration.
  • Branching Strategy: Feature branches for development, main branch for production-ready code.

Continuous Integration/Continuous Delivery (CI/CD)

  • CI/CD Pipeline: Automated testing and deployment using Azure Dev
  • Environment-Specific Pipelines:
    • Development environment: automated testing and deployment, feature branches specific.
    • UAT environment: gated workflows and manual approvals for deployment, pull requests specific.
    • Production environment: automated deployment after successful UAT environment deployment, merges to main specific.

Testing

  • Unit Testing: Automated testing for individual components and functions.
  • Integration Testing: Testing the interaction between components and services.
  • Functional Testing: End-to-end testing of application features and user scenarios.

Infrastructure as Code (IaC)

  • Azure Bicep templates are used to define and deploy infrastructure resources.
  • Infrastructure changes are version-controlled and deployed alongside application code.

Monitoring and Logging

  • Application Insights: Collects telemetry data for application performance and usage
  • Azure Monitor: Provides centralized logging and monitoring for infrastructure resources.

Deployment Triggers Across Environments

Development

  • Automated testing and deployment on push to feature branch.

UAT

  • Automated testing and deployment on successful PR to main branch.
  • Manual approvals for deployment to ensure stakeholder review and acceptance.

Production

  • Automated deployment on successful deployment to UAT environment.
  • Manual approvals for deployment to ensure readiness for end-users.

Benefits

  • Consistent deployment practices across environments.
  • Automated testing and validation for reliable releases.
  • Reduced risk of errors and issues in production.
  • Enhanced collaboration between development and operations teams.

Conclusion

The release strategy for the IE BestBank MVP application ensures efficient and reliable deployments across multiple environments. By following a structured approach to release management, we can deliver new features and updates with minimal risk and downtime.

Sequence Diagrams

The following sequence diagrams illustrate the deployment process for the IE BestBank MVP application.

1. User Account: Registration

Purpose: Outlines the steps a user takes to register an account.

User Registration Sequence Diagram

  • The user enters their details and submits the registration form.
  • The frontend sends the registration data to the backend via an API call.
  • The backend validates the data and creates a new user account in the database.
  • A confirmation message is displayed to the user upon successful registration.

2. User Account: Login

Purpose: Describes the user login process.

User Login Sequence Diagram

  • The user enters their login credentials and submits the login form.
  • The frontend sends the login data to the backend via an API call.
  • The backend validates the credentials against the database and generates a session token.
  • The user is redirected to the dashboard upon successful login.

3 and 4. User Account: Transaction

Purpose: Illustrates the user transaction process.

User Transaction Sequence Diagrams

  • The user initiates a transaction by selecting the desired action (e.g., deposit, transfer).
  • The frontend sends the transaction request to the backend via an API call.
  • The backend processes the transaction and updates the user's account balance in the database.
  • A confirmation message is displayed to the user upon successful completion.
