Site Reliability Engineering Overview

This section details IE BestBank's Site Reliability Engineering practices, incident management, and performance improvements.

SLA and corresponding SLO's and SLI's

Service Level Agreement (SLA) are agreements between a service provider and a customer that define the expected level of service. Service Level Objectives (SLOs) are specific, measurable goals that are designed to ensure that the SLA is met. Service Level Indicators (SLIs) are the metrics that are used to measure the performance of a service.

Service Level Objectives (SLO's)

The SLOs selected for this application are summarised in the following table:

Number Category Description
1 Uptime - Database & Key Vault Maintain 99% uptime for the database and Key Vault services over the monitoring period.
2 Response Time - HTTP Requests Ensure the average HTTP response time remains below 2 seconds for all requests.
3 Request Volume Tracking Track the number of HTTP requests processed hourly to ensure steady traffic flow.
4 Failed Connection Rate Limit failed database connections to less than 10% of total connection attempts.
5 Error Rate - HTTP Requests Limit failed requests (5xx errors) to less than 1% of the total number of requests.

Why These SLOs?

These SLOs are a result of collaboration with BestBank’s full-stack developers, infrastructure developers, and the cloud architect. The team has chosen these SLOs for the time being, and they’re subject to change if they don’t reflect BestBank’s preferences in terms of monitoring.

Uptime

Rationale: Maintaining availability of the database and Key Vault is important to the functionality of our application as downtime impacts the user experience and administrative operations.

Threshold: The target of 99% uptime aligns with realistic infrastructure constraints while meeting business needs.

Response Time

Rationale: Fast response times are vital for user satisfaction and operational performance. Slow responses can lead to user frustration and loss of trust.

Threshold: The 2-second threshold balances the performance demands with system capabilities.

Request Volume Tracking

Monitoring request volume assures the system can handle traffic fluctuations and prevents bottlenecks. It also helps in identifying trends in user activity.

Failed Connection Rate

Database connection failures can indicate underlying issues in the system. Limiting these failures ensures system reliability and reduces downtime risks.

Error Rate

Failed HTTP requests (5xx errors) indicate server issues. Keeping error rates below 1% ensures a seamless experience for users and meets SLA expectations.

Service Level Indicators(SLI's)

Number SLO Corresponding SLI(s)
1 99% Uptime for Database & Key Vault Uptime Percentage: The ratio of time the database and Key Vault services are available over the monitoring period.
2 Average HTTP Response Time Response Time Measurement: Tracks the average response time for HTTP requests, ensuring it remains below 2 seconds.
3 Request Volume Tracking Request Volume: Monitors the number of HTTP requests processed hourly to identify traffic patterns.
4 Failed Connection Rate Connection Reliability: Tracks the percentage of failed database connections relative to the total connection attempts.
5 Less than 1% failed requests Error Rate: Measures the percentage of failed HTTP requests (5xx errors) relative to the total number of requests.

Service Level Indicators (SLIs) Rationale

Measuring and monitoring important metrics with Azure tools is part of the SLI implementation process. Application Insights' custom metrics are used to measure response time, with queries summarizing average response time, while Azure Monitor's availability tests are used to track uptime. The connection reliability is tracked using PostgreSQL logs in Log Analytics, while the request volume is obtained from Azure Log Analytics, which groups HTTP requests by hour. Last but not least, error rates are computed to monitor the proportion of unsuccessful HTTP requests using Application Insights and Log Analytics. In order to improve SLOs and SLIs based on performance and feedback, the next steps involve validating all SLIs using these tools and performing quarterly reviews. Monitoring Strategy

The Site Reliability Engineering (SRE) team at BestBank created this document to describe the monitoring plan they used to make sure the application complies with the SLA and meets its SLOs.

Type of Monitoring

The selected monitoring strategy is focused on application monitoring and fault tolerance, which includes tracking both infrastructure-level and application-level metrics to ensure system reliability, performance, and availability. Future implementations could also include capability monitoring.

Data Collection Points

Data collection points of our monitoring strategy include telemetry data sent to our application insights by the frontend and backend of the application in addition to diagnostic settings of selected Azure resources which have been configured to send logs to our log analytics workspace.

Tool Used

Azure Workbooks with Kusto Query Language (KQL) queries are used to visualize metrics and provide actionable insights. The Workbook includes:

  • Uptime Monitoring: Visualizes the availability of the database and Key Vault with uptime percentages.
  • Latency Analysis: Displays average HTTP response times to ensure they remain within the defined threshold (SLO: <2 seconds).
  • Throughput Tracking: Shows the number of HTTP requests processed hourly to monitor traffic patterns.
  • Error Rate Monitoring: Tracks 5xx errors to ensure the error rate stays below 1%.
  • Database Connection Reliability: Monitors failed database connection attempts and ensures they remain below the 10% threshold.

Note: Additional information and metrics are also being monitored in the current version of BestBank’s workbook. While these could not directly connect to any of our selected SLOs, BestBank’s site reliability engineering team chose to keep them as a source of data for the application. This comes from the desire of BestBank being an intelligent data-driven organization.

Automation

The Azure Workbook used for the monitoring strategy has been translated into a Bicep module (implemented via Infrastructure as Code). This aligns with BestBank’s practice of employing modularization of services and resources for reusability and consistency.

This step will deploy BestBank’s workbook each time there is a code change and will ensure that the workbook does not lose the resources it is tied to (i.e., Application Insights and Log Analytics Workspace).

Incidence Response Approach

This document outlines the incident response approach implemented by BestBank’s Site Reliability Engineering (SRE) team to address cases where any of the selected Service Level Objectives (SLOs) is at risk of being violated.

Objectives of Implementing an Incident Response Strategy

  • Minimize Downtime: Ensure incidents are resolved promptly to maintain SLA compliance.
  • Real-Time Alerts: Enable rapid detection of incidents and notify the relevant team members.
  • Structured Response: Follow a defined workflow to assess, triage, and resolve incidents.
  • Continuous Improvement: Review and learn from incidents to prevent recurrence.

Alerting Workflow

For the alerting workflow, we used Logic Apps that help automate the process in addition to Slack, which is the chosen communication platform by BestBank. We created a channel for the alerts, where we configured an “API” which has a webhook URL that was utilized by the HTTP POST to Slack method in the alerting workflow. This will be further elaborated on later in this document.

Key Features

  • Recurrence Trigger: The Logic App is triggered every 5 minutes to ensure timely checks for alerts.
  • Slack Notification: Sends an HTTP POST request to the Slack Webhook URL with the alert message.
  • Customizable Alerts: The alert message can be modified to include additional details in the future, such as severity and affected services.

Incident Management Workflow

BestBank’s SRE team chose to employ the following incident management workflow:

Detect

Based on the alert rules defined, if any of the rules is satisfied, we proceed to the alerting stage.

Alert

The Logic App sends a real-time notification to the Slack channel, alerting the team about the incident.

Acknowledge

As BestBank is committed to being transparent, users will be informed about the issue and any potential outages or disruptions (possibly through a notification on the website). Future implementations could include personalized email notifications to affected users.

Investigate

The SRE team reviews diagnostic data to identify the cause of the issue. This includes:

  • Checking logs in Log Analytics for anomalies.
  • Reviewing metrics in Azure Workbooks for patterns or spikes.
  • Using tools like Application Insights for detailed performance analysis.
  • Relevant team members (e.g., developers, infrastructure engineers) are called based on the nature of the problem.

Resolve

The responsible team, in collaboration with the SRE team, works on implementing a fix. The resolution process includes:


  • Deploying changes to the development and User Acceptance Testing (UAT) environments.
  • Verifying the fix in these environments before deploying it to production.

Post-Mortem

After the incident is resolved, a post-mortem is conducted to analyze the issue and improve future responses. In this meeting, the team will analyze the problem and discuss what could be done to avoid similar problems in the future. This aligns with BestBank’s generative culture, which aims to avoid blame and encourages shared responsibility for incidents.

Future Action (if applicable)

  • Implement measures to prevent recurrence of the incident, including adjusting monitoring thresholds for earlier detection.
  • Automate resolution steps for repetitive issues.
  • Add specific dashboards or queries for improved visibility.
  • Conduct training sessions to improve response times and processes.

Automation

The Logic App and the alert rules used to implement this incident response strategy were translated into Bicep modules (implemented via Infrastructure as Code). This practice supports BestBank’s focus on modularization, reusability, and consistency in managing services and resources.

This incident response design ensures that BestBank is equipped to handle incidents effectively, minimize downtime, and continuously improve system reliability to meet user expectations and SLA commitments.

Engineering Design

Monitoring Design and Implementation

As discussed earlier, BestBank has invested in site reliability engineering. The first step was a collaboration between the SRE team with the full-stack developers, infrastructure developers, and the cloud architect to define Service Level Objectives (SLOs) and corresponding Service Level Indicators (SLIs). This step allowed BestBank to state clear performance targets such as uptime and error rates. Afterward, the site reliability engineers collaborated with infrastructure developers and full-stack developers to configure the resources to send telemetry data and diagnostic settings to Azure Application Insights and a Log Analytics Workspace.


Implementing a monitoring strategy will contribute to greater well-being for practitioners, as well as organizational and operational performance. This ensures the team can identify potential issues early (related to defining SLOs) and reduce downtime and production outages (achieved by implementing disaster recovery plans and improvement work whenever reliability targets are missed), leading to improved operational performance and thus organizational performance.

Incident Response

As discussed earlier, BestBank’s SRE team worked on implementing an incident response approach to ensure that any outages are reported before users start complaining and raising issues. The incident response design was implemented using Azure Logic Apps and alert rules defined in a Bicep module.


Our Log Analytics Workspace and Application Insights receive continuous telemetry and diagnostic data, allowing for real-time anomaly detection. The Logic App has a recurrence trigger set up to check for alerts every hour. If an alert condition is met, it notifies the designated team channel via an HTTP POST request to a Slack Webhook URL. Service Level Objectives (SLOs), including error rates, response times, and uptime, form the basis for alert definitions. Future improvements might include more thorough alert messages that include contextual details like the affected area or suggested next actions.


Important: Our Slack webhook URL is not exposed in our code; it is saved as a repository secret in the infrastructure repository and passed to the relevant modules accordingly.

Capacity Design

To ensure BestBank's application can adapt to varying workloads and maintain top performance, the following scalability and resource management strategies have been implemented:

BestBank’s Frontend

Dynamic Scaling: Configure Azure Static Web Apps to scale automatically based on real-time metrics, such as request volume and memory usage, ensuring consistent user experience during traffic spikes. This minimizes the need for manual developer intervention, enabling a self-healing strategy (scale up or scale out).


BestBank’s Backend Scaling

Efficient Deployment Pipeline: Employ Azure Container Registry (ACR) to store and distribute container images securely, enabling rapid deployments and updates.


Database Performance Management

Adaptive PostgreSQL Scaling: Configure PostgreSQL Flexible Server to scale resources automatically, ensuring uninterrupted performance during traffic surges.


Other Services Robustness

Reliable Authentication: Monitor Key Vault usage to prevent throttling during authentication peaks, ensuring secure and consistent access to secrets.

Performance Efficiency

BestBank assures that the desired performance is achieved by implementing various strategies that handle different workloads. Backend performance is improved by using efficient API designs, asynchronous calls, and caching to minimize response times. Indexing and partitioning techniques improve query execution speeds and support large datasets during high traffic periods.


Application Insights monitors application dependencies to identify potential performance bottlenecks. Traffic patterns are simulated using Azure Load Testing to refine resource configurations so that SLOs are always met under stress conditions. These methods guarantee reliability and performance.

Cost Optimization

BestBank uses a process for cost optimization to utilize resources effectively and minimize expenses. It leverages Azure Cost Management dashboards for real-time expenditure tracking and insights into usage. Budgets and alerts on expenses keep costs within predefined limits. Prepaying for reserved instances reduces operational costs for predictable workloads. Auto-scaling ensures instances are not over-provisioned, preventing unnecessary expense.

Collaboration is central to BestBank's cost optimization strategy. The team jointly reviews Azure costs, ensuring accountability and transparency in resource allocation. Resource tagging policies will allocate costs across departments and projects accurately. Thus, BestBank maintains an optimized cloud infrastructure while adhering to financial objectives.

Future Implementations: Use AI tools to find additional cost-saving opportunities and automate resource management. Improve automated systems to adjust resources and remove unused ones, saving time and money.

Reliability Design

BestBank ensures that its system remains functional, meets performance objectives, and recovers from failures by addressing data loss, downtime, and service availability.

High Availability

  • Configured Azure Traffic Manager for Regional Failover: Routes traffic between multiple regions. If a region fails, Traffic Manager redirects traffic to another region.
  • Replication of PostgreSQL: A copy of the database in another Azure region ensures seamless failover if the primary database becomes unavailable.

Resiliency

  • Stateless Services: Services avoid reliance on specific instances. External storage (e.g., Azure Redis Cache) maintains state, allowing smooth failover and scaling.
  • Retry Policies: Retry mechanisms handle temporary issues like network timeouts, ensuring critical operations (e.g., payment processing) succeed after transient failures.
  • Circuit Breakers: Temporarily halts requests to failing services, preventing system-wide disruptions and allowing affected services time to recover.

Recovery Objectives

  • Recovery Time Objective (RTO): Systems restored within 5 minutes after a failure. Example: Database restoration within 5 minutes to minimize service disruption.
  • Recovery Point Objective (RPO): Maximum acceptable data loss is 1 minute. Example: If the database fails, only up to 1 minute of data changes may be lost, preserving critical transaction data.
